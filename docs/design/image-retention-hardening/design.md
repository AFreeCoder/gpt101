# 生产镜像保留策略技术设计

日期：2026-07-15

关联需求：`docs/requirements/image-retention-hardening/requirements.md`

## 现状

生产发布流程在切换版本前，将当前容器镜像打上唯一的 `gpt101:rollback-<时间>-<提交>` 标签，同时更新 `gpt101:rollback-latest`。新镜像以 `ghcr.io/afreecoder/gpt101:sha-<完整提交号>` 拉取，发布成功后只执行通用的 dangling image 清理，因此带标签的历史镜像会一直保留。

回滚脚本的提交号回滚路径目前拼接为 `<仓库>:<提交号>`，与构建流程实际生成的 `sha-<完整提交号>` 标签不一致，需要在本次一起修正，确保清理本地镜像后仍可依赖 GHCR 回滚。

## 方案概览

新增独立脚本 `deploy/image-retention.sh`，由生产发布工作流在全部健康检查通过后调用。脚本只管理 GPT101 的本地镜像引用，并按不同镜像 ID 保留三个版本：

1. 当前 `gpt101` 容器实际使用的镜像 ID。
2. 唯一回滚标签中时间最近、且不同于当前镜像的两个镜像 ID。
3. 其余 GPT101 回滚镜像和仓库中的本地 `sha-*` 镜像均进入待清理集合。

GHCR 历史版本本次不删除，作为本地三个版本之外的长期回滚来源。

## 版本识别

- 当前版本：从正在运行的 `gpt101` 容器读取实际镜像 ID，不依赖可变标签。
- 成功历史版本：以发布前生成的唯一 `gpt101:rollback-<时间>-<提交>` 标签作为“曾成功运行”的标记，按标签中的时间倒序选择。
- 未成功上线版本：存在本地 `<镜像仓库>:sha-*` 引用，但不属于当前版本或最近两个成功版本。
- 去重键：Docker 完整镜像 ID。同一 ID 即使同时拥有 GHCR、`latest` 和多个回滚标签，也只占一个保留名额。

## 清理算法

1. 校验 Docker、镜像仓库参数和保留数量。
2. 读取当前容器镜像 ID；读取失败立即终止，不做任何删除。
3. 枚举唯一回滚标签并按时间倒序排列，选择两个不同的历史镜像 ID。
4. 枚举所有 GPT101 唯一回滚标签和指定仓库的本地 `sha-*` 引用，将不在保留集合中的镜像 ID 去重后加入待清理集合。
5. 对每个待清理镜像再次检查是否被运行中或已停止的容器引用；被引用时跳过删除并返回失败状态。
6. 只移除该镜像上的以下本地引用：
   - `gpt101:rollback-*`
   - `<指定镜像仓库>:sha-*`
7. 不按镜像 ID 强制删除，避免误删同一镜像可能存在的其他项目引用。
8. 输出当前版本、保留的历史版本、已删除引用和跳过原因。

脚本提供 `--dry-run`，用于线上执行前预览，不产生写操作。

## 发布时序

清理调用必须位于以下门禁之后：

1. PostgreSQL 健康。
2. Web 应用健康。
3. Worker 容器状态为 `running`。

任一门禁失败时，发布工作流在到达清理步骤前退出。清理本身失败时，线上新版本继续运行，但工作流返回失败并提示需要人工处理镜像占用，避免磁盘治理失败被静默忽略。

## 回滚路径

- 最近一次快速回滚：`deploy/rollback.sh image` 使用 `gpt101:rollback-latest`。
- 最近两个本地版本：通过 `deploy/rollback.sh list-images` 查找唯一回滚标签后执行镜像回滚。
- 更早版本：`deploy/rollback.sh source <完整提交号>` 从 GHCR 拉取 `sha-<完整提交号>` 后启动。
- 数据库恢复仍要求显式指定恢复后的应用镜像或提交号，不改变现有安全约束。

## 风险与控制

- 标签重复导致误计数：按完整镜像 ID 去重。
- 清理当前镜像：当前容器镜像 ID 始终先加入保留集合。
- 误删其他项目镜像：只移除两类明确的 GPT101 标签，不执行全局 `docker image prune -a`。
- 删除仍被容器引用的镜像：清理前检查所有容器，发现引用即跳过并报错。
- GHCR 版本不可用：本次不删除远端版本；发布和回滚仍使用不可变 `sha-*` 标签。
- 清理与发布并发：GitHub Actions 现有 `silicon-deploy` 并发组继续串行化自动发布。
