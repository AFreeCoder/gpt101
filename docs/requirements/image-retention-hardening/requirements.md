# 生产镜像保留策略优化需求

日期：2026-07-15

## 背景

当前每次生产发布前都会为正在运行的镜像创建回滚备份。随着发布次数增加，生产服务器本地积累了越来越多历史镜像，持续占用磁盘空间，并提高后续发布因磁盘不足而失败的风险。

## 目标

- 生产服务器本地只保留当前正在运行的版本，以及最近两个成功的生产版本。
- 只有新版本完成全部生产健康检查后，才允许清理旧镜像。
- 清理历史镜像后，最近两个成功版本仍可直接用于快速回滚。
- 更早的历史版本仍可从 GHCR 重新拉取并回滚。
- 清理行为必须可审计，能够看出保留和删除了哪些版本。

## 非目标

- 本次不删除 GHCR 中的历史镜像版本。
- 本次不调整数据库备份的七天保留规则。
- 本次不清理生产服务器上的其他项目镜像、容器、数据卷或构建缓存。
- 本次不处理 OpenClaw、Codex 或 Hermes 的磁盘清理。

## 业务规则

1. “成功的生产版本”是指曾完成生产发布并通过应用、数据库和 Worker 检查的版本。
2. 保留数量按不同镜像版本计算，不按标签数量计算；同一镜像的多个标签只算一个版本。
3. 当前正在运行的镜像必须无条件保留。
4. 发布失败或健康检查未通过时，不得执行历史镜像清理。
5. 除当前版本外，优先保留时间最近的两个成功生产版本。
6. 未成功上线、且不属于保留集合的临时拉取镜像也应被清理。
7. 如果某个待清理镜像仍被任何容器引用，不得强制删除，应明确报告并留待人工处理。

## 验收标准

- 一次成功发布完成后，正常情况下生产机本地最多保留三个不同的 GPT101 生产镜像版本。
- 最近两个成功生产版本均可通过本地回滚标签找到。
- 任一健康检查失败时，清理动作不会发生。
- 清理不会删除当前运行镜像，也不会删除非 GPT101 镜像。
- 通过提交号回滚时，能够拉取构建流程实际发布的 `sha-<完整提交号>` 镜像标签。
- 自动化测试覆盖保留三个版本、清理多余成功版本、清理未成功上线镜像和失败时不清理等关键规则。
